## 1. 引言

### 1.1 项目背景
在当前快节奏的业务（尤其是金融软件服务）中，员工的日常工作（“执行”）与其核心的岗位价值（“职责”）之间容易出现脱节。传统的`To-do List`或任务管理工具，往往只关注“做什么”（What），而忽略了“为什么做”（Why）。这导致员工可能非常忙碌，但其工作内容并未有效支撑其岗位职责的落地。同时，管理者也难以快速、准确地洞察团队的工作是否聚焦于核心目标，特别是关键任务的进展。

### 1.2 项目目的
本项目旨在开发一个“岗责驱动的周工作计划管理系统”。其核心目标是：
1.  **连接职责：** 将员工的周工作计划与其标准化的岗位职责深度绑定。
2.  **引导执行：** 通过系统智能建议，引导员工规划“有价值”的任务。
3.  **聚焦重点：** 通过“重点工作”标识，帮助团队聚焦关键任务并确保其闭环。
4.  **管理闭环：** 实现“计划-执行-复盘-辅导”的完整管理闭环，确保职责落地。

### 1.3 目标用户
系统包含三个主要用户视角：
1.  **员工 (Employee):** 系统主要使用者。用于规划、执行和复盘自己的周工作。
2.  **管理者 (Manager):** 团队主管。用于查看团队成员的周计划、负荷、完成情况，指派任务，并进行审阅和辅导。
3.  **管理员 (Administrator):** 系统配置者（如HR/PMO/信息中心）。用于定义和维护公司的“岗位-职责”模板库和组织架构。

## 2. 核心理念与流程

### 2.1 “岗责驱动”的数据模型
系统必须建立一个三层关联模型，作为所有任务的“根”：
* **岗位 (Role):** 公司的标准职位（如：销售、工程交付）。
* **职责 (Responsibility):** 该岗位被雇佣的核心价值（如：老客户关系维护）。
* **标准任务类型 (Task Type):** 为履行该职责而产生的标准动作（如：定期拜访客户）。

### 2.2 “多角色聚合”
系统必须支持一个员工（User）同时关联**多个**“岗位”（Roles）。
* **例：** `张三 = [销售经理] + [售前工程师] + [项目总监]`。
* 当员工制定计划时，系统必须聚合其所有角色的职责和任务类型，供其选择。

### 2.3 “周工作闭环”
系统必须实现以下工作流，确保任务得到跟踪：
1.  **计划 (Plan):** 员工在系统“智能建议”（基于其所有岗位职责）的引导下，创建周计划。
2.  **执行 (Execute):** 员工按计划执行，并在系统中更新任务状态。
3.  **复盘 (Review):** 周末，员工对本周计划进行复盘。
4.  **闭环 (Close-loop):**
    * **已完成：** 归档。
    * **未完成：** **强制**要求填写“未完成原因”，并选择“后续动作”（如：**[延期至下周]** 或 [取消]）。
    * **延期：** 自动滚动到下一周的“待办计划”中。

## 3. 功能规格 (Functional Specifications)

### 3.1 模块一：组织与权限中心 (管理员)
* **REQ-1.1 用户管理：** 管理员可增、删、改、查公司员工账户。
* **REQ-1.2 组织架构：** 管理员可维护公司的部门层级结构。
* **REQ-1.3 岗位关联 (核心)：** 管理员必须能够为一个员工账户关联一个或多个“岗位”（岗位来自 3.2 模块）。这是实现“多角色聚合”的基础。
* **REQ-1.4 汇报关系管理：** 系统必须支持定义用户间的“直属上级”关系（例如：在用户管理中指定其“汇报对象”）。这是实现【模块 3.5 团队视图】的数据基础。

### 3.2 模块二：岗位职责库 (管理员)
此模块是系统的“大脑”，用于定义所有“智能建议”的来源。
* **REQ-2.1 岗位库管理：** 管理员可创建、编辑、停用“标准岗位”（如：销售、研发、业务工程师等）。
* **REQ-2.2 职责定义：** 管理员可为**每一个“标准岗位”**添加多条“核心职责”（如：“销售”岗 -> “新客户开拓”职责）。
* **REQ-2.3 任务类型定义 (核心)：** 管理员可为**每一条“职责”**添加多个“标准任务类型”（如：“新客户开拓”职责 -> “销售线索跟进”任务类型）。
* **REQ-2.4 配置项停用规则：** 岗位(REQ-2.1)、职责(REQ-2.2)、任务类型(REQ-2.3) 不支持物理删除，只支持“停用”。
    * “已停用”的配置项在【创建新计划】(REQ-3.1) 时不再出现。
    * “已停用”的配置项在历史周报和统计中必须保持原样显示，以确保历史数据完整性。

### 3.3 模块三：周计划管理 (员工)
* **REQ-3.1 创建新一周计划 (引导式)：**
    * **REQ-3.1.1 智能建议：** 系统必须自动展示员工**所有关联岗位**的**所有职责**及**所有任务类型**，作为“智能建议”。
    * **REQ-3.1.2 任务来源：** 员工点击“任务类型”（如：[拜访客户 +]）可快速具体化为“本周任务”。员工也必须能够添加“非职责任务”（如：领导安排、个人临时）。
    * **REQ-3.1.3 标记重点：** 在创建/编辑任务时，员工必须可以将任务标记为 **[★ 重点工作]**。
* **REQ-3.2 延期任务处理：** 创建周计划时，系统必须自动列出“上周复盘时延期”的任务(REQ-4.4)，供员工确认是否排入本周。
* **REQ-3.3 我的仪表盘 (周视图)：**
    * **REQ-3.3.1 任务展示：** 以看板或列表形式，清晰展示本周所有任务。
    * **REQ-3.3.2 状态可见：** 任务状态需清晰可见（待办、进行中、已完成、已延期）。
    * **REQ-3.3.3 任务过滤：** 必须支持按“岗位角色”（如：只看“销售岗”任务）、“来源”（如：只看“领导安排”）以及 **[★ 只看重点]** 进行过滤。
    * **REQ-3.3.4 周中添加：** 仪表盘必须提供一个醒目的“**快速添加任务**”入口，允许员工在周中随时补录任务（并可选关联职责或标记为重点）。
* **REQ-3.4 重点任务高亮：** 在仪表盘(REQ-3.3)及所有任务列表中，“重点工作”必须有**强视觉标识**（例如星标★、高亮或专属Tag）。

### 3.4 模块四：周复盘与总结 (员工)
此模块是实现“工作闭环”的关键。
* **REQ-4.1 引导式复盘：** 系统在周五（或指定时间）提醒员工复盘。员工需逐一核对本周任务。
* **REQ-4.2 状态标记：** 员工将任务标记为 [已完成] 或 [未完成]。
* **REQ-4.3 未完成原因（强制）：** 对于标记为 [未完成] 的任务，系统**必须**要求员工从预设选项（或填写）“未完成原因”（如：客户原因、内部资源不足、优先级变更等）。
* **REQ-4.4 后续动作（强制）：** 对于 [未完成] 的任务，系统**必须**要求员工选择“后续动作”：**[延期至下周]** 或 **[取消任务]**。
* **REQ-4.5 周报自动生成：**
    * **REQ-4.5.1 周报结构：** 完成复盘后，系统自动生成一份结构化周报，内容包括：完成/未完成列表、未完成原因分析、下周计划（含延期任务）等。
    * **REQ-4.5.2 周报突出重点：** 生成的周报中，[★ 重点工作] 必须**置顶显示**或单独分组，并清晰标注其**完成状态**（如：[完成] / [未完成]）及（未完成）原因。
* **REQ-4.6 (超时默认逻辑)：** 必须定义“未复盘”的兜底逻辑。
    * **建议：** 若员工在截止时间（如：下周一09:00）前仍未完成复盘，系统自动将所有“待办/进行中”的任务标记为 [未完成-超时未复盘]，并**强制滚存**至新一周计划中（并显著标记），同时发送通知给其管理者(基于REQ-1.4)。

### 3.5 模块五：团队视图 (管理者)
* **REQ-5.1 团队仪表盘：**
    * **REQ-5.1.1 团队概览：** 管理者可查看其所有直属下属（基于 REQ-1.4）的周计划概览。
    * **REQ-5.1.2 关键指标：** 概览需包括：任务总数（负荷）、完成率、延期数、复盘状态（未提交、已提交、已审阅）。
    * **REQ-5.1.3 重点任务概览：** 仪表盘的表格视图中，必须增加“**重点任务**”列（例如：显示 `2(完成) / 3(总数)`）。
* **REQ-5.2 成员详情下钻 (审阅)：**
    * **REQ-5.2.1 详情查看：** 管理者可点击进入任意成员的“周报详情页”(REQ-4.5)。
    * **REQ-5.2.2 细节洞察：** 可查看该成员的详细任务列表、[未完成原因] 和 [后续动作]。
    * **REQ-5.2.3 审阅重点：** 过滤器中需增加 **[★ 只看重点]** 选项，以便管理者快速审阅关键任务。
    * **REQ-5.2.4 角色洞察：** 管理者可按“岗位角色”筛选下属的任务（如：查看张三本周所有“项目总监”相关的任务）。
* **REQ-5.3 审阅与反馈 (管理闭环)：**
    * **REQ-5.3.1 评论反馈：** 管理者可在下属的周报上发表“评论”或“辅导建议”。员工将收到通知。
    * **REQ-5.3.2 确认审阅：** 管理者可将周报标记为 [已审阅]。该状态同步至团队仪表盘(REQ-5.1.2)。
* **REQ-5.4 任务指派 (管理者)：**
    * **REQ-5.4.1 任务下派：** 管理者必须能够在【团队视图】中，主动为一名或多名下属**指派任务**。
    * **REQ-5.4.2 任务接收：** 被指派的任务在员工端(REQ-3.3)自动标记为“来源：领导安排”。
    * **REQ-5.4.3 指派重点：** 管理者在指派任务时，必须可以将其直接标记为 **[★ 重点工作]**。

## 4. 非功能性需求

* **REQ-NFR-1 性能：** 核心页面（仪表盘、创建计划）加载时间在标准网络环境下不得超过2秒。
* **REQ-NFR-2 易用性：** 界面设计需简洁、专业、高效。核心流程（创建计划、周复盘）必须有清晰的步骤引导。
* **REQ-NFR-3 安全性：**
    * **数据隔离：** 严格遵循权限控制。员工只能查看自己的数据；管理者只能查看其直属下属的数据。
    * **部署方式：** (根据贵公司金融客户特性) 系统需支持在客户本地化（私有化）部署，以满足数据涉密性要求。
* **REQ-NFR-4 集成性：** 应预留API接口，以便未来（二期）与公司内部OA、钉钉/企业微信集成，用于发送“复盘提醒”、“延期提醒”或“管理者评论”通知。

## 5. 附录

### 5.1 核心数据模型 (简化)
* **User** (用户表)
* **Role** (岗位库/模板表)
* **UserRoleLink** (用户-岗位关联表, 多对多)
* **Responsibility** (职责库, 关联Role)
* **TaskType** (标准任务类型库, 关联Responsibility)
* **WeeklyTask** (周计划任务表, 核心)
    * `task_id`, `user_id`, `week_num`, `title`, `status`
    * `source_type` (来源：职责, 领导安排, 个人临时)
    * `linked_task_type_id` (外键, 可空, 用于追溯职责)
    * `is_key_task` (布尔值, 默认false) - *用于标记重点任务*
    * `assigned_by_manager_id` (外键, 可空) - *用于任务指派*
* **TaskReview** (任务复盘表, 记录未完成原因、后续动作)
* **ReportComment** (周报评论表, 供管理者使用)
